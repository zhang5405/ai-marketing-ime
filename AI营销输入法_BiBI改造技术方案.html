1. 为什么选 BiBi Keyboard 而不是 Trime

❌ 旧路径：Fork Trime (GPL-3.0)

许可证GPL-3.0 → 商用必须开源全部代码
技术栈Kotlin + YAML配置 + 传统View
IME框架Trime 自有，定制受 YAML 约束
AI集成需从零开发 SSE + LLM 对接
悬浮球无（Phase 1 不包含）
语音输入Phase 1 砍掉了
总工期22-35天 + 15-23天脱GPL = 37-58天
法律风险Phase 2前必须迁移引擎，否则不可商用

✅ 新路径：基于 BiBi Keyboard (Apache 2.0)

许可证Apache 2.0 → 允许商用闭源，无传染
技术栈Kotlin 2.2 + Jetpack Compose + Material3
IME框架已有 AsrKeyboardService (InputMethodService)
AI集成已有 LlmPostProcessor → 改 Prompt 即可变营销AI
悬浮球已有，跨应用输入场景更实用
语音输入已有 sherpa-onnx 本地 + 云端 ASR
总工期19-31天（无脱GPL负担）
法律风险从Day 1零风险
核心结论

选择 BiBi Keyboard 路径，省 2-4 周工期 + 从第一天起无 GPL 法律风险。唯一新增工作是集成 librime-android 拼音引擎（这本来就是 Phase 2 的任务，现在提前做而已）。

新增收益：免费获得语音输入能力 + 悬浮球跨应用使用 + 现代化 Compose UI 框架。

4. 改造模块详解

4.1 ✅ 直接复用模块（无需改动）

悬浮球系统 (ui/floating/)

BiBi 的悬浮球已经实现了完整的跨应用输入体验：拖拽移动、权限管理、动画效果、语音触发。这是营销场景的杀手级功能——用户在任何 App 里点悬浮球就能快速使用话术。

改造点：新增一个悬浮球快捷菜单项「📋 话术」，点击弹出话术搜索悬浮窗。

IME Bridge 跨通信 (imebridge/)

ImeBridgeClient 实现了跨IME通信机制和会话管理，让你的营销键盘能与系统默认键盘协同工作。用户可以选择只使用悬浮球+话术功能，不必完全替换系统输入法。

改造点：零改动，直接复用。

剪贴板管理 (clipboard/)

剪贴板监听和管理功能，话术输入后自动清理剪贴板等场景可复用。

改造点：零改动，直接复用。

4.2 🔵 改造增强模块（修改现有代码）

IME 键盘框架 → 三Tab切换 (ime/)

BiBi 现有的 AsrKeyboardService 只有一个模式（语音输入），需要改造为三Tab切换模式：

Tab功能实现方式
⌨️ 拼音键盘标准拼音输入 + 候选词集成 librime-android，onKeyDown → RimeEngine.processKey() → 候选词列表
📋 话术面板分类Tab + 话术列表 + 一键输入新增 Compose ScriptPanelView，Room 数据源
🤖 AI 面板润色/回复/生成 + 流式输出改造 LlmPostProcessor，新增 AiPanelView

代码改动：修改 AsrKeyboardService.onCreateInputView()，返回一个包含三Tab的 Compose 容器。每个Tab切换时显示对应面板。语音输入按钮移至拼音键盘Tab内的工具栏。

LLM 后处理 → 营销AI处理器 (asr/LlmPostProcessor.kt)

BiBi 的 LlmPostProcessor 当前做的事：ASR识别结果 → LLM纠错 → 输出修正文本。本质是 "调用LLM API + 流式输出"。

改造方案：

改造项原BiBI改造后
system_prompt"请修正语音识别错误"3个营销Prompt模板：润色/回复/文案生成
LLM后端OpenAI/DeepSeek API直连经由本地FastAPI代理（密钥安全）
输入来源仅ASR语音结果3种：①ASR语音 ②手动输入文本 ③话术模板+客户消息
输出目标替换ASR结果上屏流式展示到AI面板 → 用户确认 → 上屏/复制
模式数量1个（纠错）3个（润色/自动回复/文案生成）

工作量：2-3天，主要是改Prompt模板和新增模式切换逻辑，SSE流式部分完全复用。

ASR引擎 → 瘦身保留 (asr/)

BiBI 有11个ASR引擎，你的营销场景只需要2个：

保留原因
sherpa-onnx 本地ASR离线可用，隐私友好，营销场景"口述话术"很实用
DeepSeek 云端ASR更准确，适合正式场合

砍掉：Azure、Google、Amazon、iFlytek等8个引擎实现（减少代码体积和依赖）。

工作量：1天，删除不需要的引擎实现文件和相关配置。

4.3 🟣 新增模块（从零开发）

拼音输入引擎 (librime-android 集成)

这是改造中最大的新增模块。BiBi 是纯语音输入法，没有拼音打字能力。必须集成 librime-android（BSD许可证，无GPL风险）补上拼音引擎。

集成方案

步骤工作内容预估工时
C2-1引入 librime-android AAR 到 build.gradle.kts0.5天
C2-2创建 RimeEngineBridge.kt 桥接类，封装 librime API1天
C2-3在 AsrKeyboardService.onKeyDown() 中分发按键事件：拼音模式→RIME，语音模式→ASR1天
C2-4RIME 方案配置：部署默认拼音方案 + 词典1天

// RimeEngineBridge.kt — RIME引擎桥接类
class RimeEngineBridge(context: Context) {
    private val rime = Rime(context)  // librime-android API

    fun processKey(keyCode: Int, modifiers: Int): RimeResult {
        val handled = rime.processKey(keyCode, modifiers)
        return if (handled) {
            RimeResult(
                composing = rime.getComposition(),    // 拼音组合串
                candidates = rime.getCandidates(),     // 候选词列表
                committed = rime.getCommittedText()    // 上屏文本
            )
        } else RimeResultEmpty
    }

    fun selectCandidate(index: Int) {
        rime.selectCandidate(index)
    }
}

@Composable
fun KeyboardRootView(
    currentTab: KeyboardTab,   // 拼音 | 话术 | AI
    onTabChange: (KeyboardTab) -> Unit
) {
    Column {
        // 顶部Tab栏
        KeyboardTabBar(currentTab, onTabChange)

        // 内容区域：根据Tab切换
        when (currentTab) {
            KeyboardTab.Pinyin  -> PinyinKeyboardView()    // RIME + 语音按钮
            KeyboardTab.Script  -> ScriptPanelView()        // 话术面板
            KeyboardTab.Ai      -> AiPanelView()            // AI面板
        }
    }
}

BiBI SSE 客户端优势

BiBI 已经有完整的 OkHttp SSE 流式客户端实现（在 LlmPostProcessor 中），你可以直接复用这段代码调用后端 SSE 接口，不需要从零写 SSE 客户端。Trime 方案需要从零开发这部分。

12. Phase 2 迁移路径（本地上云）

  • 🟢 省2-4周工期（4-6周 vs 7-13周）
  • 🟢 从Day 1零法律风险（Apache 2.0 vs GPL-3.0定时炸弹）
