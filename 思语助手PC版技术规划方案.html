1核心判断：能不能做？

✅
结论：可以做，且值得做。思语助手的核心竞争力在"话术库 + AI Agent + 营销场景"，而非输入法引擎本身。输入法的拼音/笔画/英文引擎是通用技术，PC 端完全可自研或集成开源方案。真正有价值的是业务层（话术库、AI 回复），这些几乎无需改动即可复用。去掉微信登录和云端存储后，整体更加轻量独立，无需部署服务端。

2架构差异：Android IME vs Windows TSF

这是做 PC 版最核心的技术认知。两个平台的输入法机制完全不同，不能"移植代码"，只能"移植设计"。

Android InputMethodService

继承 InputMethodService onCreateInputView() → 返回键盘 View onKeyDown/Up() → 处理按键 commitText() → 上屏文字 全程在 Java/Kotlin 虚拟机中运行 系统管理生命周期
Java/Kotlin View/Compose 系统进程内 标准 API

Windows TSF (Text Services Framework)

实现 COM 接口： ITfTextInputProcessorEx ITfKeyEventSink ITfCompositionSink ITfDisplayAttributeProvider 注册为 COM 服务器 (DLL) 通过 TSF Manager 与应用交互 必须用 C++ (COM ABI)
C++ / COM Direct2D / WPF 独立进程 (DLL) 底层 API

关键差异表

维度Android IMEWindows TSF影响
开发语言Kotlin / JavaC++（必须）完全重写
框架模型继承 Service 类实现 COM 接口架构不同
键盘 UIView/Compose 嵌入系统自绘窗口或 IME UIUI 重做
候选词窗口CandidatesViewITfCandidateListUIElement需适配
键位处理标准 KeyEventWM_KEYDOWN + TSF 按键事件映射不同
部署方式APK 安装MSI/EXE 安装 + 注册表注册方式不同
权限模型Android 权限系统用户态 DLL（无特殊权限）更简单
进程模型独立进程加载到目标进程（DLL）安全性更高要求
⚠️
关键认知：TSF 输入法是 COM DLL，会被加载到每一个使用输入法的目标进程地址空间（Chrome、Word、微信等）。这意味着：① 必须用原生 C++ 且不能有内存泄漏；② 崩溃会拖垮宿主进程；③ 不能在 DLL 内干"重活"，需要配合独立的 UI/Service 进程。

3PC 版整体架构设计

推荐架构：三进程模型

TSF DLL 必须轻量，业务逻辑放在独立进程，通过进程间通信（IPC）交互。这是搜狗、百度等桌面输入法的标准做法。

为什么是三进程模型？

问题如果放一起三进程方案
稳定性引擎崩溃 → 目标进程崩溃引擎崩溃，DLL 降级为基本键盘，宿主不受影响
内存每个目标进程载入完整引擎+词典(156MB)词典只在 Service 进程加载一次，所有宿主共享
性能词典 I/O 在宿主进程 UI 线程IPC 异步，词典操作在专门线程
更新DLL 被占用，无法热更新Service 进程可独立重启更新
调试输入法崩溃无法分离各进程独立日志和调试

5各模块技术方案

5.1 拼音输入引擎（最核心、最难）

拼音引擎决定了输入法的"智商"。三种方案：

方案优点缺点推荐度
A. 解析并重新实现现有引擎逻辑 完全匹配现有词典格式，体验一致 需要逆向理解 Kotlin 引擎算法（查词、组句、排序逻辑），工作量大且不保证完全一致 中等
B. 基于开源拼音引擎（推荐） 成熟稳定，社区维护，文档齐全 需要将现有词典转换为开源引擎格式，或适配 SQLCipher 读取；整句智能可能略有差异 ★★★★★
C. 从零自研拼音引擎 完全自主可控 拼音引擎是几十年积累的领域（词库、语言模型、排序算法），从零做性价比极低 不推荐
💡
推荐方案 B：开源拼音引擎
可选 librime (RIME 中州韻)，这是目前最成熟的开源中文输入法引擎，支持拼音/双拼/笔画/仓颉，有 C++ API，可加载自定义词典。思语助手的 SQLCipher 词典可以解密后转换为 RIME 格式，或通过 SQLCipher C API 直接查询后喂给引擎。RIME 也是小狼毫（Windows 版 RIME）的引擎，有成熟的 TSF 实现可参考。

5.2 TSF IME DLL（最底层）

Windows 官方 SDK 提供了 TSF 示例代码，可以在此基础上开发。

核心 COM 接口
ITfTextInputProcessorEx // 激活/停用 ITfKeyEventSink // 键盘事件 (KeyDown/KeyUp) ITfCompositionSink // 组合/候选管理 ITfDisplayAttributeProvider // 下划线/高亮样式 ITfThreadMgrEventSink // 线程焦点切换
TSF DLL 生命周期
1. 用户切换到此输入法 2. TSF Manager 加载 DLL → 调用 Activate() 3. ITfKeyEventSink.OnKeyDown() 收到按键 4. DLL 通过 IPC 发送按键到 Service 进程 5. Service 返回候选词列表 6. DLL 渲染候选窗 (Direct2D 或系统 ITfUIElement) 7. 用户选词 → ITfComposition.EndComposition() → 上屏 8. 用户切换走 → Deactivate() 9. DLL 可能被卸载 (取决于 TSF Manager 策略)
⚠️
注意事项：TSF DLL 中对宿主进程的任何阻塞操作（如同步 IPC、文件 I/O、锁等待）都会让整个目标应用卡死。所有操作必须是异步的。

5.3 键盘 UI & 候选窗

桌面输入法的键盘 UI 有两种实现方式：

方式 A：TSF UIElement（系统候选窗）方式 B：自绘悬浮窗（推荐）
实现ITfCandidateListUIElementWS_EX_LAYERED 半透明窗口 + Direct2D/GDI+
优点与系统主题一致，无需自绘完全自定义，可实现复杂 UI（Tab 切换、话术面板、图片素材）
缺点UI 功能受限，无法做话术 Tab 等复杂界面需要处理窗口层级、DPI 缩放、多屏等细节
适用纯打字输入法思语助手（营销型，需要话术/AI 面板）
💡
思语助手的核心竞争力是"键盘即营销台"，需要在候选中嵌入话术、AI 回复、图片素材等。所以必须用自绘悬浮窗方案 B，才能实现与 Android 版类似的 Tab 切换体验。

9风险与挑战

🔴 高风险
TSF 开发门槛高
TSF 文档稀少（以 MSDN 为主），社区资料少，调试困难（DLL 加载到其他进程）。国内做过商业级 TSF 输入法的团队屈指可数。建议：招有 Windows IME 经验的人，或基于 RIME 的 TSF 实现（小狼毫）做二次开发。
🔴 高风险
兼容性噩梦
输入法 DLL 会被加载到每一个进程。不同应用的 IME 支持程度差异巨大：Chrome 对 TSF 支持最好，但某些老旧企业软件、游戏、UWP 应用可能存在兼容性问题。需要大量的兼容性测试和 workaround。
🟡 中风险
杀毒软件误报
输入法 DLL 加载到其他进程的行为会被杀软视为可疑。新签发的代码签名证书需要一段时间建立信誉。建议使用 EV 代码签名证书 + 主动提交各大杀软厂商白名单。
🟡 中风险
词典格式转换精度
xiaoguandb.db 为 SQLCipher 加密，需要找到密钥才能解密。如果密钥硬编码在 DEX/Native 层中，需要深度逆向分析。解密后转为 RIME 格式可能丢失部分私有的排序/联想信息。

总结

思语助手 PC 版在技术上完全可行，去除手写、语音、VIP、分销、微信登录和云端存储后，已成为一个完全本地化的离线输入法，无需服务端即可运行。最大的挑战仍是 TSF 框架的技术门槛 和 输入法兼容性的工程深度。好消息是核心资产（词典、话术库设计、AI Agent 设计）可参考复用，真正需要从零攻坚的只有输入法框架层、桌面 UI 和本地存储。

输入方式为 拼音 + 笔画 + 英文 三种纯键盘输入；存储全部本地化（SQLite + 文件系统），无需登录、无需网络。业务层面聚焦 话术库 + AI Agent 两大核心。MVP 工作量约 1.5 人月，完整版约 3.5 人月。

建议先做一个轻量 Demo 验证 PC 端需求，再逐步投入完整开发。如果团队没有 Windows 系统编程经验，强烈建议招聘或外包有 TSF 输入法开发经验的人来主导框架层。
